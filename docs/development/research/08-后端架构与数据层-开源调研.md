## 一、需求背景与范围界定

### 1.1 PRD 原文需求摘录

根据《PRD-网文创作辅助工具-v3.0.md》技术方案与产品边界：

> **数据存储**：本地 IndexedDB + 文件导出，暂不接入云端，保护创作隐私
> **AI 能力**：DeepSeek / Claude API，长文本理解、上下文记忆、内容生成
> **上下文管理**：自定义上下文组装器，将作品数据组装为 prompt 喂给 AI
> **导出**：html2canvas + jsPDF，导出关系图、剧情树为图片/PDF
> **隐私优先**：作品数据本地存储，不上传云端 —— 网文作者对稿件敏感

性能验收标准：
- AI 对话响应时间 < 5s（常规生成））< 15s（复杂生成如剧情树）
- 可视化画布 50 个节点时拖拽流畅
- 首次加载 < 3s

## 二、本地数据存储方案调研

### 2.1 候选方案概览

#### 方案 A：Dexie.js（IndexedDB 封装）— 当前选用

- **官网**：https://dexie.org/
- **GitHub**：https://github.com/dexie/Dexie.js
- **周下载量**：~300万（npm 2026年2月数据）
- **Bundle 体积**：~65KB gzipped

**核心特性**：
- 基于 IndexedDB 的 ORM 风格封装
- 支持 TypeScript，类型安全
- 链式查询 API（类似 MongoDB/Lodash）
- 支持复合索引、范围查询
- 支持数据库版本升级与数据迁移
- 提供 `useLiveQuery` React Hook，数据变更自动重渲染
- 支持批量操作（bulkAdd/bulkPut/bulkDelete）

**代码示例**：
```typescript
import Dexie, { type Table } from 'dexie'

class WangWenDB extends Dexie {
  characters!: Table Table<Character>
  plotNodes!: Table Table<PlotNode>
  
  constructor() {
    super('wangwen-db')
    this.version(1).stores({
      characters: 'id, workId, name, [workId+name]',
      plotNodes: 'id, workId, [workId+status]',
    })
  }
}

// 查询示例
const chars = await db.characters
  .where('[workId+name]')
  .between([workId, 'A'], [workId, 'Z'])
  .toArray()

// Live Query（自动响应数据变更）
const characters = useLiveQuery(
  () => db.characters.where('workId').equals(workId).toArray(),
  [workId]
)
```

**优势**：
- 浏览器原生支持，无需额外运行时
- 存储容量大（通常为磁盘剩余空间的 50%+）
- 支持索引，查询效率高
- 事务支持，保证数据一致性
- 成熟稳定，社区活跃

**劣势**：
- IndexedDB API 本身较底层，Dexie 虽封装但仍需理解索引概念
- 复杂关联查询不如 SQL 直观（需手动组装）
- 不支持 JOIN 操作
- 单条记录大小限制约几十MB（对文本数据足够）

**适用场景**：
- 需要结构化存储的 Web 应用
- 离线优先（Offline-first）应用
- 数据量中等（GB 级别以下）

#### 方案 B：SQLite + sql.js（WASM 版本）

- **官网**：https://sql.js.org/
- **GitHub**：https://github.com/sql-js/sql.js
- **周下载量**：~50万
- **Bundle 体积**：~1MB（WASM 二进制，可缓存）

**核心特性**：
- 完整 SQLite 引擎编译为 WebAssembly
- 支持标准 SQL 语法（包括 JOIN、聚合、子查询）
- 内存数据库模式，数据以 Uint8Array 形式持久化到 IndexedDB
- 支持事务、外键、触发器

**代码示例**：
```typescript
import initSqlJs from 'sql.js'

const SQL = await initSqlJs({
  locateFile: file => `/sql.js/${file}`
})

// 创建数据库
const db = new SQL.Database()

// 建表
db.run(`
  CREATE TABLE characters (
    id TEXT PRIMARY KEY,
    work_id TEXT NOT NULL,
    name TEXT NOT NULL,
    data TEXT NOT NULL
  );
  CREATE INDEX idx_char_work ON characters(work_id);
`)

// 插入
db.run('INSERT INTO characters VALUES (?, ?, ?, ?)', 
  [id, workId, name, JSON.stringify(data)])

// 查询
const result = db.exec(`
  SELECT c.*, COUNT(r.id) as relation_count
  FROM characters c
  LEFT JOIN relations r ON c.id = r.source_id
  WHERE c.work_id = ?
  GROUP BY c.id
`, [workId])
```

**优势**：
- 完整 SQL 支持，复杂查询能力强
- 支持 JOIN、聚合、窗口函数
- 数据完整性约束（外键、CHECK）
- 成熟的查询优化器

**劣势**：
- WASM 初始化耗时（100-500ms 首次加载）
- Bundle 体积大（1MB+）
- 内存模式：大数据集占用主内存
- 持久化需手动序列化到 IndexedDB
- 不支持多标签页同步（每个标签页独立内存实例）

**适用场景**：
- 需要复杂 SQL 查询的应用
- 数据关系复杂，大量 JOIN 操作
- 团队熟悉 SQL 开发

#### 方案 C：SQLite + OPFS（Origin Private File System）

- **文档**：https://sqlite.org/wasm/doc/trunk/persistence.md
- **技术**：官方 SQLite WASM + OPFS 持久化

**核心特性**：
- SQLite 官方 WASM 构建
- 直接写入 OPFS（Origin Private File System）
- 支持同步 I/O（在 Web Worker 中）
- 持久化透明，无需手动序列化

**优势**：
- 官方支持，稳定性高
- 直接文件系统持久化，性能优于 IndexedDB 中介
- 支持大容量存储

**劣势**：
- 同步 I/O 必须在 Web Worker 中运行，架构复杂
- 浏览器支持有限（Chrome/Edge 支持较好，Safari 部分支持）
- 需要额外 Worker 线程管理
- 开发调试复杂度增加

---

#### 方案 D：RxDB（Reactive Database）

- **官网**：https://rxdb.info/
- **GitHub**：https://github.com/pubkey/rxdb
- **周下载量**：~15万

**核心特性**：
- 基于 RxJS 的响应式数据库
- 支持多种存储后端（IndexedDB、OPFS、SQLite、内存）
- 支持实时同步（可对接 CouchDB、GraphQL、WebRTC）
- 提供 React/Vue/Angular 绑定
- 支持数据加密

**优势**：
- 响应式查询，数据变更自动推送
- 存储后端可切换（IndexedDB → OPFS → SQLite）
- 内置同步能力（未来可扩展多设备）
- 支持数据压缩和加密

**劣势**：
- 学习曲线陡峭（RxJS + 文档型数据库概念）
- Bundle 体积较大
- 对当前项目而言功能过剩（暂不需要同步）
- 社区相对较小

#### 方案 E：localStorage + 内存缓存（极简方案）

**核心特性**：
- 仅使用 localStorage 存储 JSON 字符串
- 运行时加载到内存操作
- 定期或手动持久化

**优势**：
- 实现极简，零依赖
- 同步 API，代码简单

**劣势**：
- 容量限制（通常 5-10MB）
- 同步阻塞主线程
- 无索引，查询需全量遍历
- 不适合大数据量

**适用场景**：
- 配置存储
- 极小数据量（（<1MB）

### 2.3 推荐方案

**主推荐：Dexie.js（方案 A）**

决策理由：
1. **当前项目已选用**，团队已有实践基础
2. **查询能力足够**：网文创作工具的数据查询以简单过滤和 ID 查找为主，极少需要复杂 JOIN

### 2.4 Dexie 最佳实践与代码规范

#### 数据库设计规范

```typescript
// src/db/schema.ts
import Dexie, { type Table } from 'dexie'
import type { Work, Character, PlotNode, RelationEdge, WorkSystem, Idea, ChatMessage } from '@/types'

export class WangWenDB extends Dexie {
  // === 作品表 ===
  works!: Table Table<Work>
  
  // === 角色表 ===
  characters!: Table Table<Character>
  
  // === 剧情节点表 ===
  plotNodes!: Table Table<PlotNode>
  
  // === 关系边表 ===
  relations!: Table Table<RelationEdge>
  
  // === 体系表 ===
  systems!: Table Table<WorkSystem>
  
  // === 灵感便签表 ===
  ideas!: Table<Idea>
  
  // === 对话历史表（按作品隔离）===
  chatMessages!: Table Table<ChatMessage>
  
  // === 伏笔追踪表 ===
  foreshadowings!: Table Table<Foreshadowing>
  
  // === 规则库表 ===
  rules!: Table Table<Rule>

  constructor() {
    super('wangwen-db')
    
    // v1: 初始版本
    this.version(1).stores({
      works: 'id, name, updatedAt',
      characters: 'id, workId, name, status, [workId+name], [workId+status], [workId+updatedAt]',
      plotNodes: 'id, workId, parentId, status, [workId+status], [workId+parentId], [workId+updatedAt]',
      relations: 'id, workId, sourceId, targetId, [workId+sourceId], [workId+targetId], [workId+type]',
      systems: 'id, workId, name, [workId+name]',
      ideas: 'id, workId, status, category, [workId+status], [workId+category]',
      chatMessages: 'id, workId, role, timestamp, [workId+timestamp]',
      foreshadowings: 'id, workId, status, nodeId, [workId+status], [workId+nodeId]',
      rules: 'id, workId, systemId, severity, [workId+systemId]',
    })
  }
}
```

#### 索引设计原则

| 索引类型 | 适用场景 | 示例 |
|----------|----------|------|
| 单字段索引 | 按字段等值查询 | `workId` |
| 复合索引 | 多字段联合查询 | `[workId+status]` |
| 复合范围索引 | 联合查询+排序 | `[workId+updatedAt]` |

**索引设计规则**：
1. 所有表必须包含 `workId` 索引（多作品隔离）
2. 高频查询字段组合建复合索引
3. 避免过多索引（影响写入性能）
4. 索引字段顺序：等值查询字段在前，范围查询字段在后

#### 批量操作规范

```typescript
// ✅ 正确：批量插入使用单事务
await db.transaction('rw', db.characters, async () => {
  await db.characters.bulkAdd(characters)
})

// ❌ 错误：逐条插入，性能极差
for (const char of characters) {
  await db.characters.add(char) // 每次新建事务！
}

// ✅ 正确：批量更新
await db.characters.bulkPut(updatedCharacters)

// ✅ 正确：条件批量删除
await db.characters
  .where('workId')
  .equals(workId)
  .delete()
```

#### 方案 C：自定义 Fetch + EventSource

**核心特性**：
- 直接使用浏览器 Fetch API 调用 AI 接口
- 手动解析 SSE 流
- 零依赖（除类型定义外）

**代码示例**：
```typescript
async function* streamChat(messages: Message[]): AsyncGenerator<string> {
  const response = await fetch('https://api.deepseek.com/v1/chat/completions', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': `Bearer ${apiKey}`,
    },
    body: JSON.stringify({
      model: 'deepseek-chat',
      messages,
      stream: true,
    }),
  })

  const reader = response.body!.getReader()
  const decoder = new TextDecoder()
  let buffer = ''

  while (true) {
    const { done, value } = await reader.read()
    if (done) break
    
    buffer += decoder.decode(value, { stream: true })
    const lines = buffer.split('\n')
    buffer = lines.pop() || ''

    for (const line of lines) {
      if (line.startsWith('data: ')) {
        const data = line.slice(6)
        if (data === '[DONE]') return
        try {
          const json = JSON.parse(data)
          yield json.choices[0]?.delta?.content || ''
        } catch { }
      }
    }
  }
}
```

**优势**：
- 零依赖，Bundle 最小
- 完全控制请求和解析逻辑
- 无框架绑定，任何项目可用
- 调试透明

**劣势**：
- 需手动处理所有边界情况
- 无内置重试、错误处理
- 多供应商支持需自行封装
- 开发成本高

### 3.4 AI 上下文管理方案

#### 问题定义

网文创作工具的 AI 助手需要维护的上下文：
- 当前作品全部角色档案（可能 50-100 个角色，每个 500-2000 字）
- 当前作品全部剧情节点（可能 100-500 个节点，每个 200-1000 字）
- 当前作品全部关系数据
- 当前作品全部体系设定
- 当前作品全部灵感便签
- 用户最近的 20 轮对话历史

**问题**：全部上下文可能超过 100K tokens，远超模型上下文窗口（DeepSeek-chat 支持 64K，但实际有效使用应控制在 32K 以内）。

#### 解决方案：分层上下文压缩

```typescript
// src/ai/context-builder.ts

interface ContextLayer {
  priority: number      // 1-10，数字越小优先级越高
  content: string       // 组装后的文本
  tokens: number        // 估算的 token 数
}

class ContextBuilder {
  private maxTokens = 24000 // 预留 16K 给回复
  
  async build(workId: string, userMessage: string): Promise<string> {
    const layers: ContextLayer[] = [
      await this.getSystemPrompt(),           // P0: 系统提示词
      await this.getRecentChatHistory(workId), // P1: 最近对话
      await this.getActivePlotNodes(workId),   // P2: 活跃剧情节点
      await this.getMainCharacters(workId),    // P3: 主要角色
      await this.getRelevantIdeas(workId),     // P4: 相关灵感
      await this.getSystemRules(workId),       // P5: 体系规则
      await this.getFullCharacters(workId),    // P6: 全部角色（摘要）
    ]
    
    let totalTokens = 0
    const selectedLayers: string[] = []
    
    for (const layer of layers.sort((a, b) => a.priority - b.priority)) {
      if (totalTokens + layer.tokens <= this.maxTokens) {
        selectedLayers.push(layer.content)
        totalTokens += layer.tokens
      } else {
        // 尝试压缩该层
        const compressed = await this.compressLayer(layer, this.maxTokens - totalTokens)
        if (compressed) {
          selectedLayers.push(compressed)
          totalTokens += this.estimateTokens(compressed)
        }
        break
      }
    }
    
    return selectedLayers.join('\n\n')
  }
  
  private async compressLayer(layer: ContextLayer, budget: number): Promise<string | null> {
    // 使用 AI 对上下文进行摘要压缩
    // 保留关键信息，舍弃细节
    return await aiSummarize(layer.content, budget)
  }
}
```

#### Token 估算策略

```typescript
// src/ai/token-counter.ts

/**
 * 快速 Token 估算（无需精确，用于预算控制）
 * 中文：1 token ≈ 1 个汉字
 * 英文：1 token ≈ 0.75 个单词
 */
export function estimateTokens(text: string): number {
  const chineseChars = (text.match(/[\u4e00-\u9fff]/g) || []).length
  const englishWords = (text.match(/[a-zA-Z]+/g) || []).length
  const otherChars = text.length - chineseChars - englishWords
  
  return Math.ceil(chineseChars + englishWords * 0.75 + otherChars * 0.5)
}
```

### 4.1 高频查询场景分析

| 查询场景 | 频率 | 当前实现 | 优化建议 |
|----------|------|----------|----------|
| 按作品 ID 查询所有角色 | 高 | `where('workId').equals(id)` | 已优化，使用单索引 |
| 按作品+状态查询剧情节点 | 高 | `where('[workId+status]').equals([id, status])` | 复合索引已覆盖 |
| 角色名称搜索（模糊） | 中 | 全表遍历 + JS filter | 使用 Dexie `startsWith` 或全文索引 |
| 关系网络查询（某角色的所有关系） | 高 | `where('sourceId').equals(id)` | 双索引：sourceId + targetId |
| 跨表关联查询（角色+其关系数） | 中 | 多次查询手动组装 | 预计算缓存或 Dexie `hook` |
| 时间范围查询（最近更新） | 中 | `where('updatedAt').above(date)` | 添加 updatedAt 索引 |

  const works = await db.works.toArray()

  }

    size: JSON.stringify(backup).length,
  })

    .orderBy('timestamp')
    .reverse()
    .offset(MAX_BACKUPS)

  await db.backups.bulkDelete(oldBackups.map(b => b.id))
}

export function startAutoBackup() {
  setInterval(createBackup, BACKUP_INTERVAL)

### 7.1 API Key 安全存储

```typescript
// src/utils/crypto.ts

const ENCRYPTION_KEY_NAME = 'wangwen:encryption-key'

/**
 * 使用 Web Crypto API 加密敏感数据
 */
export async function encryptData(plainText: string): Promise<string> {
  const key = await getOrCreateKey()
  const iv = crypto.getRandomValues(new Uint8Array(12))
  
  const encoded = new TextEncoder().encode(plainText)
  const encrypted = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv },
    key,
    encoded
  )
  
  const combined = new Uint8Array(iv.length + encrypted.byteLength)
  combined.set(iv)
  combined.set(new Uint8Array(encrypted), iv.length)
  
  return btoa(String.fromCharCode(...combined))
}

export async function decryptData(cipherText: string): Promise<string> {
  const key = await getOrCreateKey()
  const combined = Uint8Array.from(atob(cipherText), c => c.charCodeAt(0))
  
  const iv = combined.slice(0, 12)
  const encrypted = combined.slice(12)
  
  const decrypted = await crypto.subtle.decrypt(
    { name: 'AES-GCM', iv },
    key,
    encrypted
  )
  
  return new TextDecoder().decode(decrypted)
}

async function getOrCreateKey(): Promise<CryptoKey> {
  const stored = localStorage.getItem(ENCRYPTION_KEY_NAME)
  
  if (stored) {
    const keyData = Uint8Array.from(atob(stored), c => c.charCodeAt(0))
    return crypto.subtle.importKey('raw', keyData, 'AES-GCM', false, ['encrypt', 'decrypt'])
  }
  
  const key = await crypto.subtle.generateKey(
    { name: 'AES-GCM', length: 256 },
    true,
    ['encrypt', 'decrypt']
  )
  
  const exported = await crypto.subtle.exportKey('raw', key)
  const exportedArray = new Uint8Array(exported)
  localStorage.setItem(ENCRYPTION_KEY_NAME, btoa(String.fromCharCode(...exportedArray)))
  
  return key
}
```
