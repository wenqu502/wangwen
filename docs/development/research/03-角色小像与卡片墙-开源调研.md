## 一、需求背景与范围界定

### 1.1 PRD 原文需求摘录

根据《PRD-网文创作辅助工具-v3.0.md》第 5.1 节「角色小像」：

> **核心理念**：用户用一句话描述角色，AI 生成完整档案。
>
> **AI 对话式创建**：
> - 用户描述角色特征，AI 生成完整档案（姓名、外貌、性格、背景、创伤、目标等）
> - AI 自动生成姓名（可修改）
> - 生成后询问是否需要形象图
>
> **AI 对话式修改**：
> - 自然语言修改角色属性
> - 连锁更新：修改后同步更新相关描述（如经典台词、背景故事）
> - 一致性检测提醒：修改后提醒扫描已写章节检测冲突
>
> **AI 形象生成**：
> - 调用文生图 API 生成角色形象
> - 一次生成 4 张风格备选
> - 支持风格调整指令
>
> **人设一致性校验（AI 自动）**：
> - 修改档案后自动扫描已写章节标记冲突
> - 撰写新章节时实时检测角色言行是否符合档案
> - 提交章节时生成一致性报告

**角色档案数据结构**：

```json

  "id": "char_001",
  "name": "顾沉",

  "tags": ["男主", "外科医生", "高冷", "偏执"],

    "keywords": ["疏离", "细心", "偏执", "控制欲"],
```

根据《PRD-网文创作辅助工具-v2.0.md》补充界面需求：

> - 角色卡片墙：网格/列表/看板三种视图
> - 角色筛选：按身份/阵营/性别/状态筛选
> - 角色搜索：实时搜索角色名和标签
> - 角色排序：按创建时间/出场顺序/名字排序
> - 点击角色节点：展开角色小像侧边栏，快速查看档案

### 3.1 TipTap —— 现代富文本编辑器框架

| 项 | 内容 |
|----|------|
| 官网 | https://tiptap.dev |
| GitHub | https://github.com/ueberdosis/tiptap |
| Stars | 28k+ |
| 最新版本 | v3.x（2024 年发布） |
| 技术栈 | ProseMirror + TypeScript + React/Vue |
| 开源协议 | MIT |
| 定位 | 无头（Headless）富文本编辑器框架 |
| 维护团队 | überdosis（德国团队） |
| 更新频率 | 每周发版，非常活跃 |

**核心能力详解**

1. **无头架构（Headless）**：
   - 不提供默认 UI，只提供编辑逻辑和 API
   - 开发者完全控制编辑器外观和交互
   - 与 Tailwind CSS / 自定义设计系统完美契合

2. **扩展系统（Extensions）**：
   - 核心扩展：Document、Paragraph、Text、Heading、List、Blockquote 等
   - 标记扩展：Bold、Italic、Strike、Highlight、Link 等
   - 节点扩展：Image、Table、CodeBlock 等
   - 功能扩展：History、Placeholder、CharacterCount、Collaboration 等
   - 自定义扩展 API：可开发私有扩展

3. **React 集成**：
   - `@tiptap/react` 提供 `useEditor` Hook
   - `EditorContent` 组件渲染编辑区域
   - `BubbleMenu` / `FloatingMenu` 悬浮菜单组件
   - 命令链式调用：`editor.chain().focus().toggleBold().run()`

4. **数据格式**：
   - 默认输出 JSON（ProseMirror Document）
   - 支持 HTML 输入/输出
   - 支持 Markdown 输入/输出（通过扩展）

**与织文需求的匹配度分析**

| 织文需求 | TipTap 支持情况 | 实现复杂度 |
|----------|----------------|------------|
| 角色档案富文本编辑 | 完全支持，无头架构可完全自定义样式 | 低 |
| 多字段编辑（外貌/背景/创伤） | 每个字段一个 TipTap 编辑器实例 | 低 |
| 关键词标签高亮 | `Highlight` 扩展 + 自定义颜色 | 低 |
| 经典台词列表 | `BulletList` / `OrderedList` 扩展 | 低 |
| 占位符提示 | `Placeholder` 扩展 | 低 |
| 字数统计 | `CharacterCount` 扩展 | 低 |
| 撤销/重做 | `History` 扩展 | 低 |
| 图片插入（形象图） | `Image` 扩展 | 低 |
| 协作编辑（未来） | `Collaboration` 扩展（基于 Yjs） | 中 |

**优缺点**

| 优点 | 缺点 |
|------|------|
| 无头架构，UI 完全可控 | 需要自行实现所有 UI 控件 |
| React 集成优秀 | 包体积中等（~200KB gzip） |
| 扩展系统强大，生态丰富 | 学习曲线中等 |
| TypeScript 原生支持 | 中文文档相对较少 |
| 社区极其活跃 | — |
| MIT 协议 | — |

**对织文的借鉴点**

| 借鉴项 | 具体应用 |
|--------|----------|
| 无头架构 | 角色档案编辑器完全自定义样式，匹配织文设计规范 |
| BubbleMenu | 选中文本后弹出格式化菜单（加粗/斜体/高亮） |
| Placeholder | 空字段显示提示文字（如「点击输入角色外貌描述...」） |
| CharacterCount | 角色简介字数限制提示 |
| JSON 输出 | 角色档案字段直接存储为 ProseMirror JSON |

### 3.2 Slate.js —— 完全可定制的富文本编辑器框架

| 项 | 内容 |
|----|------|
| 官网 | https://www.slatejs.org |
| GitHub | https://github.com/ianstormtaylor/slate |
| Stars | 30k+ |
| 最新版本 | v0.110+ |
| 技术栈 | React + TypeScript |
| 开源协议 | MIT |
| 定位 | 完全可定制的富文本编辑器框架 |
| 维护团队 | Ian Storm Taylor + 社区 |
| 更新频率 | 持续维护 |

**核心能力详解**

1. **数据驱动架构**：
   - 编辑器状态为纯 JSON 对象（Slate Document）
   - 所有变更通过 `Transforms` API 操作
   - 完全可预测的状态管理

2. **自定义渲染**：
   - `renderElement`：自定义块级节点（段落、标题、列表、引用等）
   - `renderLeaf`：自定义行内标记（加粗、斜体、颜色等）
   - 完全控制渲染输出

3. **插件系统**：
   - 通过 `withXXX` 高阶函数扩展编辑器行为
   - 社区插件：slate-history、slate-react、slate-hyperscript 等

4. **React 深度集成**：
   - `Slate` 上下文组件
   - `Editable` 编辑区域组件
   - `useSlate` Hook 获取编辑器实例

**与织文需求的匹配度分析**

| 织文需求 | Slate.js 支持情况 | 实现复杂度 |
|----------|------------------|------------|
| 角色档案富文本编辑 | 完全支持 | 中 |
| 多字段编辑 | 每个字段一个 Slate 编辑器 | 中 |
| 关键词标签高亮 | 自定义 Leaf 渲染 | 中 |
| 经典台词列表 | 自定义 Element 渲染 | 中 |
| 占位符提示 | 需自行实现 | 中 |
| 字数统计 | 需自行计算 | 中 |
| 撤销/重做 | `slate-history` 插件 | 低 |
| 图片插入 | 自定义 Image Element | 中 |

**优缺点**

| 优点 | 缺点 |
|------|------|
| 数据模型简洁清晰 | API 相对底层，开发工作量更大 |
| React 集成原生 | 插件生态不如 TipTap 丰富 |
| 完全可控的渲染 | 很多功能需自行实现 |
| 类型安全（TypeScript） | 文档分散，学习成本高 |
| 社区庞大 | 版本迭代快，API 变动 |

**对织文的借鉴点**

| 借鉴项 | 具体应用 |
|--------|----------|
| 数据驱动模型 | 角色档案字段统一用 Slate JSON 存储 |
| 自定义 Element | 特殊字段区块（如「经典台词」带引用样式） |
| 自定义 Leaf | 关键词自动高亮（如性格关键词加粗着色） |

**核心能力详解**

   - `@dnd-kit/sortable`：排序列表/网格
   - `@dnd-kit/utilities`：工具函数

   - Pointer Sensor（鼠标/触摸）

   - `useSortable` Hook
   - `SortableContext` 组件
   - 支持列表和网格排序
   - 支持多容器拖拽（跨列拖拽）

4. **可访问性**：
   - 完整的键盘操作支持
   - ARIA 属性自动管理
   - 屏幕阅读器友好

| 键盘无障碍 | 内置 Keyboard Sensor | 低 |

| 优点 | 缺点 |

| 可访问性优秀 | — |
| 排序功能完善 | — |
| TypeScript 原生 | — |

**对织文的借鉴点**

| 借鉴项 | 具体应用 |

### 3.5 react-window / react-virtualized —— 虚拟滚动列表

| 项 | 内容 |
|----|------|
| 官网 | https://github.com/bvaughn/react-window |
| GitHub | https://github.com/bvaughn/react-window |
| Stars | 15k+ |
| 最新版本 | v1.8.x |
| 技术栈 | React |
| 开源协议 | MIT |
| 定位 | 轻量级虚拟滚动列表 |
| 维护团队 | Brian Vaughn（React 核心团队成员） |
| 更新频率 | 稳定维护 |

**核心能力详解**

1. **虚拟滚动原理**：
   - 只渲染可视区域内的元素
   - 滚动时动态更新渲染的元素
   - DOM 节点数恒定，不随数据量增加

2. **组件类型**：
   - `FixedSizeList`：等高的列表项
   - `VariableSizeList`：不等高的列表项
   - `FixedSizeGrid`：等高的网格项
   - `VariableSizeGrid`：不等高的网格项

3. **性能优势**：
   - 10,000 条数据也能流畅滚动
   - 内存占用恒定

**与织文需求的匹配度分析**

| 织文需求 | react-window 支持情况 | 实现复杂度 |
|----------|----------------------|------------|
| 大量角色卡片滚动 | `FixedSizeGrid` / `VariableSizeGrid` | 低 |
| 列表视图滚动 | `FixedSizeList` | 低 |
| 网格视图滚动 | `FixedSizeGrid` | 低 |
| 卡片高度不一致 | `VariableSizeGrid` + `getItemSize` | 中 |

**优缺点**

| 优点 | 缺点 |
|------|------|
| 性能优异，大数据量流畅 | 需要预先知道每项尺寸 |
| 包体积极小（~5KB gzip） | 复杂布局（如瀑布流）支持有限 |
| React 核心团队维护 | 网格布局的拖拽集成较复杂 |
| 稳定可靠 | — |

**对织文的借鉴点**

| 借鉴项 | 具体应用 |
|--------|----------|
| FixedSizeGrid | 角色卡片墙网格视图虚拟滚动 |
| FixedSizeList | 角色列表视图虚拟滚动 |
| itemData | 传递角色数据到渲染项 |

## 四、综合评分与选型决策

### 4.1 评分矩阵

**注：角色卡片墙模块涉及多个技术点，此处按「核心依赖库」维度评分。**

| 维度（权重） | TipTap | Slate.js | @dnd-kit | react-beautiful-dnd | react-window |
|-------------|--------|----------|----------|---------------------|--------------|
| 技术栈匹配度（25%） | 5 | 5 | 5 | 4 | 5 |
| 功能完备度（25%） | 5 | 4 | 4 | 3 | 3 |
| 自定义能力（20%） | 5 | 5 | 4 | 3 | 3 |
| 性能表现（15%） | 4 | 4 | 4 | 4 | 5 |
| 社区活跃度（10%） | 5 | 4 | 4 | 3 | 4 |
| 许可证（5%） | 5 | 5 | 5 | 5 | 5 |
| **加权总分** | **4.85** | **4.40** | **4.15** | **3.55** | **3.75** |

**评分细则**

- **TipTap（4.85/5）**：
  - 富文本编辑领域综合最优
  - 无头架构与织文自定义设计需求完美匹配
  - 扩展生态最丰富，未来可扩展协作编辑
  - React 集成优秀

- **Slate.js（4.40/5）**：
  - 数据模型更清晰
  - 但开发工作量大，插件生态不足
  - 作为 TipTap 的备选方案

- **@dnd-kit（4.15/5）**：
  - 现代 React 拖放首选
  - 网格排序、多容器拖拽、可访问性全面
  - 轻量且活跃维护

- **react-beautiful-dnd（3.55/5）**：
  - 维护模式扣分
  - 不支持网格布局
  - 仅在看板模式有优势

- **react-window（3.75/5）**：
  - 虚拟滚动性能最优
  - 但功能单一，仅解决滚动性能
  - 作为性能优化手段使用

### 4.2 选型结论

**富文本编辑：TipTap（主选）/ Slate.js（备选）**

理由：
1. TipTap 无头架构与织文设计规范完全兼容
2. 扩展系统丰富，占位符、字数统计、历史记录全部内置
3. React 集成优秀，开发效率高
4. 社区活跃，v3 版本持续迭代

**拖拽排序：@dnd-kit（主选）**

理由：
1. 现代 React Hooks API
2. 网格排序 + 多容器拖拽完全覆盖需求
3. 可访问性优秀
4. 轻量且活跃维护

**虚拟滚动：react-window（辅助）**

理由：
1. 角色数量超过 100 时启用虚拟滚动
2. 包体积极小，性能提升显著
3. 与 @dnd-kit 需测试兼容性

## 六、Zustand Store 设计

```typescript
// src/modules/character/store.ts

import { create } from 'zustand'
import { immer } from 'zustand/middleware/immer'
import { persist } from 'zustand/middleware'
import {
  Character,
  CharacterFilter,
  CharacterSortBy,
  CharacterRole,
  CharacterStatus,
  ViewMode,
  ConsistencyConflict,
  CharacterImage,
} from './types'

interface CharacterStore extends CharacterState {
  // CRUD 操作
  addCharacter: (character: Character) => void
  updateCharacter: (id: string, updates: Partial<Character>) => void
  removeCharacter: (id: string) => void
  setCharacters: (characters: Character[]) => void

  // 筛选排序
  setFilter: (filter: Partial<CharacterFilter>) => void
  setSortBy: (sortBy: CharacterSortBy) => void
  toggleSortOrder: () => void
  setViewMode: (mode: ViewMode) => void

  // 选中
  selectCharacter: (id: string | null) => void

  // 冲突
  setConflicts: (conflicts: ConsistencyConflict[]) => void
  resolveConflict: (conflictId: string) => void

  // 形象生成
  setGeneratingImage: (isGenerating: boolean) => void
  setImageGenerationProgress: (progress: number) => void
  addCharacterImage: (characterId: string, image: CharacterImage) => void

  // 计算属性
  getFilteredCharacters: () => Character[]
  getCharacterById: (id: string) => Character | undefined
  getCharactersByFaction: (faction: string) => Character[]

  // 导入导出
  exportToJSON: () => string
  importFromJSON: (json: string) => void
}

export const useCharacterStore = create<CharacterStore>()(
  immer(
    persist(
      (set, get) => ({
        // 初始状态
        characters: [],
        filter: {
          roles: [],
          factions: [],
          statuses: [],
          searchQuery: '',
        },
        sortBy: CharacterSortBy.CREATED_AT,
        sortOrder: 'desc',
        viewMode: ViewMode.GRID,
        selectedCharacterId: null,
        conflicts: [],
        isGeneratingImage: false,

        // CRUD
        addCharacter: (character) =>
          set((state) => {
            state.characters.push(character)
          }),

        updateCharacter: (id, updates) =>
          set((state) => {
            const index = state.characters.findIndex((c) => c.id === id)
            if (index !== -1) {
              Object.assign(state.characters[index], updates, {
                updatedAt: new Date().toISOString(),
              })
            }
          }),

        removeCharacter: (id) =>
          set((state) => {
            state.characters = state.characters.filter((c) => c.id !== id)
            if (state.selectedCharacterId === id) {
              state.selectedCharacterId = null
            }
          }),

        setCharacters: (characters) =>
          set((state) => {
            state.characters = characters
          }),

        // 筛选排序
        setFilter: (filter) =>
          set((state) => {
            Object.assign(state.filter, filter)
          }),

        setSortBy: (sortBy) =>
          set((state) => {
            state.sortBy = sortBy
          }),

        toggleSortOrder: () =>
          set((state) => {
            state.sortOrder = state.sortOrder === 'asc' ? 'desc' : 'asc'
          }),

        setViewMode: (mode) =>
          set((state) => {
            state.viewMode = mode
          }),

        // 选中
        selectCharacter: (id) =>
          set((state) => {
            state.selectedCharacterId = id
          }),

        // 冲突
        setConflicts: (conflicts) =>
          set((state) => {
            state.conflicts = conflicts
          }),

        resolveConflict: (conflictId) =>
          set((state) => {
            state.conflicts = state.conflicts.filter((c) => c.id !== conflictId)
          }),

        // 形象生成
        setGeneratingImage: (isGenerating) =>
          set((state) => {
            state.isGeneratingImage = isGenerating
          }),

        setImageGenerationProgress: (progress) =>
          set((state) => {
            state.imageGenerationProgress = progress
          }),

        addCharacterImage: (characterId, image) =>
          set((state) => {
            const char = state.characters.find((c) => c.id === characterId)
            if (char) {
              char.images.push(image)
            }
          }),

        // 计算属性
        getFilteredCharacters: () => {
          const { characters, filter, sortBy, sortOrder } = get()

          let result = [...characters]

          // 搜索过滤
          if (filter.searchQuery) {
            const query = filter.searchQuery.toLowerCase()
            result = result.filter(
              (c) =>
                c.name.toLowerCase().includes(query) ||
                c.tags.some((t) => t.toLowerCase().includes(query)) ||
                c.aliases.some((a) => a.toLowerCase().includes(query))
            )
          }

          // 角色过滤
          if (filter.roles.length > 0) {
            result = result.filter((c) => filter.roles.includes(c.role))
          }

          // 阵营过滤
          if (filter.factions.length > 0) {
            result = result.filter((c) => c.faction && filter.factions.includes(c.faction))
          }

          // 状态过滤
          if (filter.statuses.length > 0) {
            result = result.filter((c) => filter.statuses.includes(c.status))
          }

          // 排序
          result.sort((a, b) => {
            let comparison = 0
            switch (sortBy) {
              case CharacterSortBy.NAME:
                comparison = a.name.localeCompare(b.name)
                break
              case CharacterSortBy.CREATED_AT:
                comparison = new Date(a.createdAt).getTime() - new Date(b.createdAt).getTime()
                break
              case CharacterSortBy.UPDATED_AT:
                comparison = new Date(a.updatedAt).getTime() - new Date(b.updatedAt).getTime()
                break
              case CharacterSortBy.FIRST_APPEARANCE:
                comparison = (a.firstAppearance || '').localeCompare(b.firstAppearance || '')
                break
            }
            return sortOrder === 'asc' ? comparison : -comparison
          })

          return result
        },

        getCharacterById: (id) => {
          return get().characters.find((c) => c.id === id)
        },

        getCharactersByFaction: (faction) => {
          return get().characters.filter((c) => c.faction === faction)
        },

        // 导入导出
        exportToJSON: () => {
          return JSON.stringify(get().characters, null, 2)
        },

        importFromJSON: (json) =>
          set((state) => {
            try {
              state.characters = JSON.parse(json)
            } catch (e) {
              console.error('Import failed:', e)
            }
          }),
      }),
      {
        name: 'wangwen-characters',
        partialize: (state) => ({
          characters: state.characters,
          filter: state.filter,
          sortBy: state.sortBy,
          sortOrder: state.sortOrder,
          viewMode: state.viewMode,
          conflicts: state.conflicts,
        }),
      }
    )
  )
)
```

```tsx
import React from 'react'
import { useEditor, EditorContent } from '@tiptap/react'
import StarterKit from '@tiptap/starter-kit'
import Placeholder from '@tiptap/extension-placeholder'
import Highlight from '@tiptap/extension-highlight'

interface TipTapEditorProps {
  content: string
  onChange: (html: string) => void
  placeholder?: string

export const TipTapEditor: React.FC<TipTapEditorProps> = ({
  content,
  onChange,
  placeholder = '开始输入...',
}) => {

    extensions: [
      StarterKit,
      Placeholder.configure({ placeholder }),
      Highlight.configure({ multicolor: true }),

    onUpdate: ({ editor }) => {
      onChange(editor.getHTML())
    },
  })

  if (!editor) return null

  return (
    <div className="tiptap-editor">
      <div className="editor-toolbar">
        <button
          className={editor.isActive('bold') ? 'is-active' : ''}
          onClick={() => editor.chain().focus().toggleBold().run()}
        >
          加粗
        </button>
        <button
          className={editor.isActive('italic') ? 'is-active' : ''}
          onClick={() => editor.chain().focus().toggleItalic().run()}
        >
          斜体
        </button>
        <button
          className={editor.isActive('highlight') ? 'is-active' : ''}
          onClick={() => editor.chain().focus().toggleHighlight().run()}
        >
          高亮
        </button>
      </div>

    </div>
  )
```

## 九、性能优化策略

| 优化点 | 策略 | 预期效果 |
|--------|------|----------|
| 大量卡片渲染 | 角色数 > 100 时启用 react-window 虚拟滚动 | 滚动帧率 60fps |
| 图片加载 | 头像懒加载 + 渐变占位 | 首屏加载速度提升 |
| 搜索过滤 | 100ms debounce + 使用计算属性缓存 | 搜索无卡顿 |
| 筛选状态 | URL query 同步，刷新后状态保持 | 用户体验提升 |
| Drawer 动画 | CSS transform 动画，避免重排 | 打开动画 300ms 流畅 |
| 编辑器实例 | 非活动 Tab 的编辑器销毁实例 | 内存占用减少 |
| 数据持久化 | IndexedDB 分表存储（角色/冲突/图片） | 读写性能优化 |
| 形象图缓存 | Base64 缓存到 IndexedDB | 二次打开即时显示 |

## 十、风险分析

| 风险项 | 影响 | 概率 | 缓解措施 |
|--------|------|------|----------|
| TipTap 与 Tailwind 样式冲突 | 中 | 低 | 使用 TipTap 无头架构，完全自定义样式 |
| 大量角色卡片渲染卡顿 | 高 | 中 | 虚拟滚动 + 图片懒加载 |
| 形象图 API 不稳定 | 中 | 中 | 支持多 API 源切换，本地缓存兜底 |
| 角色详情 Drawer 内容过多 | 低 | 中 | Tab 分页，按需加载 |
| @dnd-kit 与虚拟滚动兼容 | 中 | 低 | 测试验证，必要时降级为非虚拟滚动 |
| 人设一致性检测延迟高 | 中 | 低 | 后台异步检测，不阻塞用户操作 |

## 十二、文生图服务方案

| 方案 | 说明 | 优点 | 缺点 | 推荐阶段 |
|------|------|------|------|----------|
| Pollinations.ai | 免费开源文生图 API | 免费，无需 Key，国内可访问 | 质量一般，不稳定 | 开发期 |
| SiliconFlow（硅基流动） | 国内 API 平台，支持 FLUX | 国内稳定，FLUX 质量高 | 需注册 Key，按量计费 | 生产期 |
| Stable Diffusion 本地部署 | 本地运行 SD 模型 | 完全免费，隐私安全 | 需显卡资源，部署复杂 | 私有化部署 |
| Midjourney API | 商业 API | 质量最高 | 贵，国内访问困难 | 高端用户 |
| DALL-E 3 API | OpenAI 商业 API | 质量高，语义理解强 | 贵，需海外支付 | 备选 |

**推荐方案**：
- **开发期**：Pollinations.ai（零成本快速验证）
- **生产期**：SiliconFlow（国内稳定 + FLUX 高质量）
- **私有化**：本地 Stable Diffusion（保护创作隐私）

```typescript
/** AI 创建角色提示词 */
export const createCharacterPrompt = (description: string) => `
你是一位专业的小说角色设计师。请根据用户的描述，生成一个完整的角色档案。

用户描述：${description}

请生成以下字段的 JSON（不要输出 markdown 代码块，直接输出 JSON）：
{
  "name": "角色姓名（根据描述推断，富有特色）",
  "aliases": ["别名1", "别名2"],
  "tags": ["身份标签", "性格标签"],
  "appearance": "外貌描述（200字以内）",
  "personality": {
    "keywords": ["性格关键词3-5个"],
    "surface": "表面性格（100字）",
    "inner": "内心性格（100字）",
    "stressResponse": "应激反应（50字）"
  },
  "background": "背景故事（300字以内）",
  "trauma": "内心创伤（可选）",
  "goals": "角色目标（可选）",
  "arc": "角色弧光（可选）",
  "quotes": ["经典台词1", "经典台词2"],
  "abilities": ["能力1", "能力2"]
}

要求：
- 姓名要符合角色设定风格
- 性格要有层次感，避免扁平
- 背景故事要有冲突和张力
- 经典台词要有记忆点
`

${chapterContent}

检测要求：
1. 角色言行是否符合性格设定

3. 角色关系是否与档案矛盾
4. 角色背景是否被违背

如果发现不一致，输出：
- 不一致的具体位置
- 不一致的类型（性格/能力/关系/背景）
```
