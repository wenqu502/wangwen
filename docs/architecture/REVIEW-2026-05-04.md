## 三、性能优化维度

### 🔴 P0-001: Store 全量替换导致引用雪崩

**问题描述**:
`setCharacters`、`setNodes`、`setEdges` 等批量操作使用 Immer 的 `produce` 进行全量替换：

```typescript
setCharacters: (list) =>
  set((state) => {
    state.characters = {}  // 全量替换！
    for (const c of list) state.characters[c.id] = c
  }),
```

这会导致所有订阅了 `characters` 的组件触发重渲染，即使数据实际上没有变化。

**风险等级**: 🔴 **高**

**具体建议**:
1. 使用 Zustand 的 `shallow` 比较或自定义 selector 来避免不必要的重渲染
2. 批量更新时，先 diff 再更新，只修改变化的部分
3. 对于列表组件，使用 `React.memo` + 自定义比较函数

**代码示例**:

```typescript
// ❌ 当前做法：任何 set 都触发全量重渲染
const characters = useCharacterStore((s) => s.characters) // 订阅整个 Record

// ✅ 优化做法：使用 selector + shallow 比较
import { shallow } from 'zustand/shallow'

const characterList = useCharacterStore(
  (s) => Object.values(s.characters),
  shallow
)

// ✅ 更优做法：组件只订阅自己需要的数据
const characterNames = useCharacterStore(
  (s) => Object.values(s.characters).map((c) => ({ id: c.id, name: c.name })),
  shallow
)
```

### 🟡 P1-004: AI 上下文无 Token 截断策略

**问题描述**:
`ChatPanel.tsx` 中 `messages.slice(-10)` 仅按消息条数截断，未按 Token 数量计算。长消息很容易超过 DeepSeek 的上下文窗口（默认 64K）。

**风险等级**: 🟡 **中**

**具体建议**:
1. 引入简单的 Token 估算（中文 ≈ 1.5 token/字）
2. 当估算 Token 超过阈值（如 50K）时，对早期消息做摘要压缩
3. 保留 system prompt 和最近 N 条完整消息，中间历史做摘要

**代码示例**:

```typescript
function estimateTokens(text: string): number {
  // 粗略估算：中文 1.5 token/字，英文 0.25 token/字
  const cnCount = (text.match(/[\u4e00-\u9fa5]/g) || []).length
  const enCount = text.length - cnCount
  return Math.ceil(cnCount * 1.5 + enCount * 0.25)
}

function trimMessages(messages: ChatMessage[], maxTokens: number): ChatMessage[] {
  let total = 0
  const result: ChatMessage[] = []
  // 从后往前遍历，保留最新消息
  for (let i = messages.length - 1; i >= 0; i--) {
    const tokens = estimateTokens(messages[i].content)
    if (total + tokens > maxTokens) break
    total += tokens
    result.unshift(messages[i])
  }
  return result
}
```

## 四、安全分析维度

### 🔴 P0-002: VITE_DEEPSEEK_API_KEY 完全暴露于浏览器端

**问题描述**:
API Key 通过 `import.meta.env.VITE_DEEPSEEK_API_KEY` 注入，在浏览器中可通过 DevTools → Network → 查看请求头，或在 Sources 中搜索 API Key 明文。

**风险等级**: 🔴 **高**

**具体建议**:

**方案 A（推荐）: 自建代理服务器**
1. 搭建轻量级后端代理（Cloudflare Workers / Vercel Edge / 自建 Node.js）
2. 前端调用代理接口，代理在服务端注入 API Key
3. 代理层可增加请求频率限制、内容审核

```typescript
// 代理端点示例 (Cloudflare Workers)
export default {
  async fetch(request: Request, env: Env) {
    const body = await request.json()
    const response = await fetch('https://api.deepseek.com/v1/chat/completions', {
      method: 'POST',
      headers: {
        'Authorization': `Bearer ${env.DEEPSEEK_API_KEY}`, // 服务端密钥，不暴露
        'Content-Type': 'application/json',
      },
      body: JSON.stringify(body),
    })
    return response
  },
}
```

**方案 B（短期过渡）: 用户自行输入 API Key**
1. 不在 `.env` 中硬编码，改为在 UI 设置面板中让用户输入自己的 Key
2. Key 存入 `localStorage`（仍有暴露风险，但比源码硬编码好）
3. 提示用户"API Key 仅存储在本地浏览器中"

**方案 C（最小改动）: 环境变量 + 构建时剥离**
1. 使用非 VITE_ 前缀的环境变量（需后端配合）
2. 当前纯前端方案无法完全解决，只能降低暴露面

**推荐优先级**: 方案 A > 方案 B > 方案 C

### 🔴 P0-004: IndexedDB 数据无加密

**问题描述**:
用户创作内容（小说设定、角色信息、剧情等）以明文存储在浏览器 IndexedDB 中，任何有本地文件访问权限的程序（或浏览器扩展）都可读取。

**风险等级**: 🔴 **高**

**具体建议**:
1. 使用 `crypto.subtle` API 对敏感字段加密存储
2. 密钥由用户密码派生（PBKDF2），服务端不存储密钥
3. 非敏感字段（如 UI 状态）可不加密

**代码示例**:

```typescript
// 加密工具封装
async function deriveKey(password: string, salt: Uint8Array): Promise<CryptoKey> {
  const encoder = new TextEncoder()
  const keyMaterial = await crypto.subtle.importKey(
    'raw', encoder.encode(password), 'PBKDF2', false, ['deriveKey']
  )
  return crypto.subtle.deriveKey(
    { name: 'PBKDF2', salt, iterations: 100000, hash: 'SHA-256' },
    keyMaterial,
    { name: 'AES-GCM', length: 256 },
    false,
    ['encrypt', 'decrypt']
  )
}

async function encryptData(data: string, key: CryptoKey): Promise<{ iv: string; cipher: string }> {
  const iv = crypto.getRandomValues(new Uint8Array(12))
  const encoder = new TextEncoder()
  const encrypted = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv },
    key,
    encoder.encode(data)
  )
  return {
    iv: Array.from(iv).join(','),
    cipher: btoa(String.fromCharCode(...new Uint8Array(encrypted))),
  }
}

// 在 Dexie 存储前加密敏感字段
class WangWenDB extends Dexie {
  async saveCharacter(char: Character, password: string) {
    const key = await deriveKey(password, this.salt)
    const encrypted = await encryptData(JSON.stringify(char), key)
    return this.characters.put({ ...char, _encrypted: encrypted })
  }
}
```

**注意**: 加密会增加 CPU 开销，建议仅加密敏感字段（如 `background`、`personality.inner` 等），而非全部数据。

### 🟡 P1-007: 中国大陆网络稳定性

**问题描述**:
DeepSeek API 在国内可用，但用户网络环境复杂（公司防火墙、运营商劫持等），可能导致 API 调用失败。

**风险等级**: 🟡 **中**

**具体建议**:

2. 提供离线模式：AI 功能不可用时，本地功能仍可正常使用
3. 错误提示明确区分"网络问题"和"API 问题"

**代码示例**:

```typescript
async function fetchWithRetry(
  fn: () => Promise<T>,
  maxRetries = 3,
  baseDelay = 1000
): Promise<T> {
  let lastError: Error
  for (let i = 0; i < maxRetries; i++) {
    try {
      return await fn()
    } catch (err) {
      lastError = err as Error
      if (i < maxRetries - 1) {
        await new Promise((r) => setTimeout(r, baseDelay * Math.pow(2, i)))
      }
    }
  }
  throw new Error(`请求失败（已重试 ${maxRetries} 次）: ${lastError.message}`)
}
```

## 五、数据存储效率维度

### 🔴 P0-005: 无数据库版本迁移策略

**问题描述**:
当前 `this.version(1).stores(...)` 无任何迁移逻辑。当 schema 变更（如新增字段、修改索引）时，旧版本数据库无法自动升级，可能导致：
1. 用户数据丢失
2. `VersionError` 导致应用崩溃
3. 用户被迫清除浏览器数据

**风险等级**: 🔴 **高**

**具体建议**:
1. 立即设计版本升级策略
2. 每次 schema 变更递增版本号
3. 使用 Dexie 的 `upgrade` 回调处理数据迁移

**代码示例**:

```typescript
class WangWenDB extends Dexie {
  constructor() {
    super('wangwen-db')

    // v1: 初始版本
    this.version(1).stores({
      works: 'id, name, createdAt',
      characters: 'id, workId, name, [workId+name], createdAt',
      // ...
    })

    // v2: 新增角色字段 + 修改索引
    this.version(2)
      .stores({
        characters: 'id, workId, name, [workId+name], [workId+status], createdAt',
        // 新增索引
      })
      .upgrade((tx) => {
        // 迁移数据：给旧数据添加新字段默认值
        return tx.table('characters').toCollection().modify((char) => {
          char.status = char.status || 'alive'
          char.powerLevel = char.powerLevel || 0
        })
      })

    // v3: 新增作品标签功能
    this.version(3)
      .stores({
        works: 'id, name, *tags, createdAt', // * = 多值索引
      })
      .upgrade((tx) => {
        return tx.table('works').toCollection().modify((work) => {
          work.tags = work.tags || []
        })
      })
  }
}
```

**关键规则**: 版本号只能递增，不能回退。Dexie 会自动按顺序执行所有升级。

### 🟡 P1-008: Dexie 冗余索引

**问题描述**:
当前索引设计：

```typescript
characters: 'id, workId, name, [workId+name], createdAt'
```

- `id` 是主键，自动索引
- `workId` 单字段索引
- `[workId+name]` 复合索引

**问题**: 如果查询总是同时用 `workId` 和 `name`，`workId` 单字段索引是冗余的（复合索引的前缀可以覆盖单字段查询）。

**风险等级**: 🟡 **中**

**具体建议**:
1. 分析实际查询模式，移除冗余索引
2. 复合索引遵循"最左前缀原则"
3.  Dexie 索引会增加写入开销和存储空间

**优化后索引设计**:

```typescript
// 查询模式分析：
// 1. 按作品查角色列表: where('workId').equals(...)
// 2. 按作品+名称查重: where([workId, name]).equals([...])
// 3. 按创建时间排序: reverse().sortBy('createdAt')

characters: 'id, [workId+name], createdAt'
// 移除冗余的 workId 单索引，复合索引 [workId+name] 的前缀 workId 可覆盖查询模式 1
```

### 🟡 P1-010: Character.relations 双源数据风险

**问题描述**:
`Character` 类型包含 `relations: RelationEdge[]`，同时 `RelationEdge` 单独存储在 `relations` 表中。这导致同一关系数据存在两个来源，可能不一致。

**风险等级**: 🟡 **中**

**具体建议**:
1. **方案 A（推荐）**: 删除 `Character.relations`，所有关系查询统一走 `relations` 表
2. **方案 B**: 如果为了性能需要冗余，建立触发器机制，确保修改时同步更新

### 🟡 P1-011: PlotNode 双向引用维护成本

**问题描述**:
`PlotNode` 包含 `parentIds` 和 `childIds` 双向引用，新增/删除节点时需要同时维护两个方向。

**风险等级**: 🟡 **中**

**具体建议**:
1. 使用单方向引用（只存 `parentIds`），`childIds` 通过查询动态计算
2. 如果必须双向引用，在 Store action 中封装维护逻辑，禁止直接修改

### 🟡 P1-012: context-builder 串行全量加载

**问题描述**:
`buildWorkContext` 需要同时加载角色、剧情、关系、体系等全部数据，随着数据量增长会线性变慢。

**风险等级**: 🟡 **中**

**具体建议**:
1. 按模块懒加载上下文（用户问角色时只加载角色数据）
2. 对已完成章节做摘要缓存
3. 使用 Web Worker 在后台组装上下文，不阻塞 UI

### 🟡 P1-013: 存储限额无监控

**问题描述**:
未监控 IndexedDB 使用量，用户可能在不知情的情况下达到浏览器存储上限。

**风险等级**: 🟡 **中**

**具体建议**:
1. 定期调用 `navigator.storage.estimate()` 检查剩余空间
2. 超过阈值（如 80%）时提示用户导出/清理数据
3. 提供"存储管理"面板，显示各模块数据量

**代码示例**:

```typescript
async function checkStorageQuota() {
  if ('storage' in navigator && 'estimate' in navigator.storage) {
    const { usage, quota } = await navigator.storage.estimate()
    if (usage && quota) {
      const percent = Math.round((usage / quota) * 100)
      if (percent > 80) {
        console.warn(`存储使用量: ${percent}%，建议清理数据`)
        // 触发 UI 提示
      }
      return { usage, quota, percent }
    }
  }
  return null
}
```

### 🟡 P1-014: 图片存储策略未定义

**问题描述**:
`Character.images` 当前设计为字符串数组，但未明确存储的是 URL、Base64、还是 Blob 引用。

**风险等级**: 🟡 **中**

**具体建议**:

| 存储方案 | 优点 | 缺点 | 适用场景 |
|----------|------|------|----------|
| URL 引用 | 不占用本地空间 | 依赖外部服务，可能失效 | 网络图片、头像 |
| Base64 | 完全离线可用 | 体积增加 33%，IndexedDB 性能差 | 小图标、缩略图 |
| IndexedDB Blob | 完全离线，原生支持 | 占用存储配额 | 用户上传的图片 |
| File System API | 不占用 IndexedDB 配额 | 兼容性差 | 大文件存储（实验性） |

**推荐策略**:
1. 小图片（< 100KB）存 Base64 在 Dexie
2. 大图片存为 Blob，URL 使用 `URL.createObjectURL()`
3. 提供图片压缩功能（canvas 压缩到指定尺寸）
4. 定期清理未使用的 ObjectURL

### 🟢 P2-006: 导入/导出功能缺失

**问题描述**:
用户数据完全存储在本地，无导出/导入功能。浏览器数据清除或换设备时，所有创作内容丢失。

**风险等级**: 🟢 **低**

**具体建议**:
1. 实现 JSON 导出（全部数据 + 元数据）
2. 实现 JSON 导入（版本校验 + 数据迁移）
3. 可选：加密导出（用户密码保护）

**代码示例**:

```typescript
async function exportAllData(): Promise<Blob> {
  const data = {
    version: 2,
    exportedAt: new Date().toISOString(),
    works: await db.works.toArray(),
    characters: await db.characters.toArray(),
    plotNodes: await db.plotNodes.toArray(),
    relations: await db.relations.toArray(),
    systems: await db.systems.toArray(),
    ideas: await db.ideas.toArray(),
  }
  const json = JSON.stringify(data, null, 2)
  return new Blob([json], { type: 'application/json' })
}

async function importAllData(file: File): Promise<void> {
  const text = await file.text()
  const data = JSON.parse(text)

  // 版本校验
  if (data.version < 2) {
    throw new Error('数据版本过旧，无法导入')
  }

  // 清空现有数据
  await db.transaction('rw', db.tables, async () => {
    await Promise.all(db.tables.map((t) => t.clear()))
    await db.works.bulkAdd(data.works)
    await db.characters.bulkAdd(data.characters)
    // ...
  })
}
```

### 🟢 P2-007: 单表无分页查询

**问题描述**:
所有 `toArray()` 调用都会加载整张表的数据到内存，数据量大时可能导致内存溢出。

**风险等级**: 🟢 **低**

**具体建议**:
1. 列表展示使用 `offset()` + `limit()` 分页
2. 搜索功能使用 Dexie 的 `filter()` + 索引查询
3. 大数据量统计使用 `count()` 而非 `toArray().length`
